# VidGnost GPU 多进程执行清单

## 背景说明

这次优化的目标，不是继续在主进程里做“模型对象卸载 + empty_cache + 猜显存有没有真释放”，而是把当前实际参与主分析链路的本地重计算阶段，改造成统一的独立 worker 进程执行模型。

这里有一个非常关键的边界需要先说清楚：

当前项目里，真正已经随仓库稳定交付、并且已经实际接入主分析链路的本地重计算模型，核心就是 Whisper 转写链路。其它本地模型配置项虽然已经在模型目录、配置中心、下载管理或后续扩展位里出现，但它们并没有全部在当前主分析链路里形成“可稳定启用、可验证、可交付”的本地 GPU 推理闭环。

因此，这次执行清单的“完成”标准，不是把所有未来可能接入的本地模型都强行接完，而是把当前已经真实参与分析阶段、并且必须优先稳定的本地 GPU 执行路径，重构成一套干净、可扩展、可验证的多进程调度架构，并把旧的进程内 LRU 卸载逻辑清掉。

换句话说，本次交付的重点是：

1. 用统一的 GPU 运行时租约机制，替换掉旧的组件级 LRU 驱逐思路。
2. 把 Whisper 的阶段 C 转写执行改造成独立 worker 进程。
3. 保持现有阶段持久化、暂停/继续、阶段性重试能力不退化。
4. 让“阶段结束即退出进程并回收显存”成为当前本地主链路的正式执行模型。
5. 给后续 embedding、rerank、vlm、以及真正交付的本地 LLM 接入同一套 worker 协议预留干净扩展点，而不是继续堆叠主进程内缓存和卸载胶水逻辑。

这个方向比“主进程里保活多个模型实例，内存不够时再按 LRU 淘汰”更适合 VidGnost 当前的产品形态。原因很直接：

- 当前主分析链路是阶段串行，不是多个模型高频交错抢占 GPU。
- 分析任务的核心诉求是稳定跑完、可恢复、可持久化，而不是追求多模型常驻带来的极小切换收益。
- Windows + CUDA + 多个推理库混用时，进程内显存释放往往并不具备足够的确定性。
- 进程退出是目前最确定、最容易验证、也最不容易长成后期维护灾难的显存回收方式。

所以，本次执行会明确采用下面这条原则：

`哪个阶段需要本地重计算，就拉起对应 worker 进程；阶段执行期间由统一 GPU 租约独占运行；阶段结果及时持久化；阶段结束后退出 worker 进程，依靠进程退出回收显存。`

## 本次执行范围

本次执行范围限定为“当前已实际随项目交付并参与主分析链路的本地 GPU 重计算路径”，具体包括：

- 阶段 C 的 Whisper 转写执行
- GPU 运行时租约与任务调度
- 阶段产物持久化与恢复兼容
- 旧的 LRU 驱逐逻辑与相关冗余代码清理
- 相关 OpenSpec 与项目文档同步

本次不把以下内容作为已完成交付承诺：

- 还未真正接入主分析阶段的 embedding GPU 推理
- 还未真正接入主分析阶段的 rerank GPU 推理
- 还未真正接入主分析阶段的 vlm GPU 推理
- 需要基于 `Ollama` 管理的本地 LLM / Embedding / Rerank / VLM GPU 正式交付能力，并与当前 worker 调度框架形成统一边界

这些能力后续会复用本次建立的 worker 协议与 GPU 调度框架接入，但不把尚未成熟的路径硬塞进这轮执行范围。

## 目标架构

本次落地后的主分析链路执行模型如下：

1. 主进程负责任务编排、状态流转、阶段产物持久化、SSE 事件分发、暂停/取消/恢复控制。
2. GPU 重计算阶段通过统一的 worker 子进程执行。
3. 当前正式接入 worker 的阶段为阶段 C Whisper 转写。
4. 主进程在阶段进入前申请 GPU 租约，确保同一时刻只有一个重 GPU worker 占用执行槽。
5. worker 通过结构化事件流向主进程回传分段进度、chunk 完成、错误和完成事件。
6. 主进程在收到 chunk 完成事件后立即写入阶段 checkpoint，保证中途失败后可以基于已完成 chunk 继续。
7. 阶段结束后 worker 立即退出，通过进程退出回收显存，不再依赖主进程内模型卸载。

## 执行清单

### 1. 文档与边界确认

- 新增本执行清单文档到 `docs` 根目录。
- 在文档中明确：本次交付聚焦当前已实际参与主分析链路的本地 GPU 阶段，不把尚未成熟的本地模型占位能力包装成已完成能力。
- 在文档中明确：本次正式废弃“组件级 LRU 驱逐作为主显存管理手段”的方案。

### 2. GPU 运行时调度器收口

- 将现有 `ModelRuntimeManager` 的核心职责收敛为“独占租约管理”，不再承担组件级 LRU 驱逐职责。
- 保留统一的 GPU 租约入口，继续为任务阶段提供等待时间统计与活动快照能力。
- 删除不再需要的驱逐计划、跨组件驱逐和相关解释性日志逻辑。

### 3. Whisper 阶段 C 改造为独立 worker 进程

- 新增 GPU stage worker 入口模块。
- 新增主进程到 worker 的结构化请求/事件协议。
- 阶段 C 改为在单个 worker 进程中顺序处理所有未完成 audio chunk，而不是主进程内直接持有 Whisper 模型实例。
- worker 将分段事件流式回传给主进程。
- worker 将 chunk 完成事件回传给主进程，由主进程立即进行 checkpoint 持久化。
- worker 进程完成后退出，由进程退出完成显存回收。

### 4. 取消、暂停、恢复与失败恢复兼容

- 当任务取消或暂停时，主进程必须能够终止当前 worker 进程。
- 已持久化 chunk 不得丢失。
- 恢复任务时继续复用 `C/transcript/chunk-XXXX.json`、`index.json`、`full.txt` 与任务记录中的 transcript 状态。
- 只有未完成 chunk 会重新进入 worker。

### 5. 日志与观测收口

- 阶段日志改为围绕“GPU 执行槽等待”“worker 启动”“worker 退出”“checkpoint 写入”等真实用户可理解事件输出。
- 不再输出旧的 LRU 驱逐日志。
- 保留对阶段等待时间、worker 使用次数和阶段耗时的度量。

### 6. OpenSpec 同步

- 更新 `transcription-pipeline` 相关 spec，明确阶段 C 使用独立 worker 进程执行并在 chunk 级完成后持久化。
- 更新 `llm-runtime-config` 与相关描述，避免继续把当前实现表述成依赖组件级缓存驱逐。
- 同步 active change 与 baseline specs，保证规格与实现一致。

### 7. 验证与清理

- 运行后端测试。
- 运行 OpenSpec 检查与 spec 同步检查。
- 检查本次改造涉及的冗余逻辑、旧注释、无效状态字段与临时文件。
- 清理已经失效的 LRU 驱逐相关代码和测试。

## 完成标准

当以下条件同时满足时，本次清单视为执行完成：

1. 阶段 C Whisper 转写已经通过独立 worker 进程执行。
2. worker 在阶段结束或任务中断时能够退出，主进程不再依赖组件级 LRU 卸载去回收显存。
3. 已完成 chunk 的 checkpoint 仍然可以在暂停/恢复/失败后继续复用。
4. 旧的 LRU 驱逐主逻辑已经从代码中移除。
5. OpenSpec 与项目文档已同步。
6. 测试与检查脚本通过。
7. 项目中的本次改造遗留冗余逻辑和临时文件已经清理。
