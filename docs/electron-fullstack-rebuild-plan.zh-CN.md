# VidGnost Electron 全栈功能缝合与重构方案（Linear + Sentry + Raycast）

> 状态：方案文档（仅规划，不涉及代码变更）  
> 适用范围：`F:\in-house project\VidGnost`  
> 目标：将现有系统重构为 Electron + FastAPI 一体化工作台，完成前后端功能缝合与视觉信息架构改版

---

## 1. 重构目标

本次重构目标不是“把网页套个壳”，而是完成以下四类升级：

1. 运行形态升级：从浏览器工作台升级为桌面原生工作台（Electron）。
2. 交互与视觉升级：全面替换当前布局、排版、组件风格，建立“Linear 主风格 + Sentry 状态语义 + Raycast 命令交互”的设计语言。
3. 产品表达升级：以“视频分析 + 证据问答双主线（A/B/C/D/E + 子阶段）”为信息中心，构建更直观、更高信任感的任务驱动 UI。
4. 后端能力升级：将 `Local-VQA` 的 Hybrid 检索、RAG 问答、证据追踪能力融合进当前后端主链路。

## 2. 范围与边界

### 2.1 本次重构范围

- 前端运行容器：Vite Web -> Electron（主进程 + 渲染进程 + preload）
- UI 信息架构：页面结构、导航模式、面板布局、交互路径
- 视觉系统：色彩、字体、间距、组件样式、动效规范
- 前端工程组织：模块分层、状态流、IPC 封装、桌面能力接入
- 后端功能缝合：吸收 `Local-VQA` 的 Hybrid 检索、RAG 问答、证据引用与白盒追踪工作流
- 后端接口与数据：补齐 `analyze/search/chat/chat-stream/traces` 契约与数据结构兼容策略

### 2.2 明确不在本次范围

- 历史数据结构强制迁移（仅做兼容，不破坏存量）
- 模型算法层重写（本次只做流程编排与接口融合，不重训模型）
- 直接复制 `Local-VQA` 全仓库代码（采用“按能力映射实现”，不是整仓拷贝）

## 3. 当前系统架构摘要（重构基线）

前端当前是 React + Vite + TypeScript Web 应用，主入口和核心视图集中在：

- `frontend/src/App.tsx`
- `frontend/src/components/workbench-*.tsx`
- `frontend/src/hooks/use-*.ts`

后端当前是 FastAPI 任务流水线，已具备阶段执行与 SSE 输出能力，但尚未完整吸收 `Local-VQA` 的检索问答链路。

当前核心能力已具备：

- SSE 实时任务流
- 阶段日志与产物展示
- 配置面板 / 历史任务 / 导出链路
- 本地后端服务拉起与基础接口调用

对照 `Local-VQA` 目标链路后，仍存在明显缺口：

1. 阶段视图仍是 A/B/C/D 叙事，缺少 `E（RAG 问答生成）` 的一等公民入口。
2. 尚无“问答会话 + 证据引用 + 时间锚点跳转 + 原图预览”的一体化工作区。
3. 检索可观测性不足，缺少 Dense/Sparse/RRF/Rerank 的结果对照与调试面板。
4. `trace_id` 级别的过程追踪没有直观呈现，不利于定位问答质量问题。
5. 后端缺少统一的 Hybrid 检索编排与 RAG 上下文构建契约，接口语义尚未稳定。
6. `query_template` 历史字段兼容策略未制度化，存在旧数据行为不一致风险。

当前主要问题（从桌面产品目标看）：

1. 布局仍偏“网页面板拼接”，非桌面生产力应用心智。
2. 信息密度高时，视觉层级不够稳定（阶段、日志、产物争夺注意力）。
3. 窗口级能力缺位（原生菜单、系统级快捷键、文件拖放体验、窗口状态管理）。
4. 前端与后端的本地运行关系由用户手工维护，不符合“桌面应用一体化交付”预期。

## 4. 目标架构（Electron + FastAPI）

## 4.1 进程分层

- `main` 进程：窗口、菜单、协议、生命周期、自动更新、后端守护进程管理
- `preload`：受控 IPC 桥，白名单 API 暴露给渲染层
- `renderer`：React UI（纯展示与交互，不直接触碰 Node 能力）

## 4.2 后端协同模式（推荐）

采用“桌面壳 + 本地后端服务”模式：

1. Electron 启动时检测本地后端健康检查（端口从配置读取，默认兼容 `8000/8765`）
2. 未就绪则由主进程拉起后端（开发态 `uv run`，生产态打包可执行）
3. 就绪后渲染层使用本地 API（保持当前接口协议不变）

优势：最小化后端改造，保留现有任务与存储体系。

## 4.3 Local-VQA 接口语义兼容（全栈约束）

重构阶段应把以下接口视为目标能力面，而不是“可选增强”：

- `POST /api/analyze`：视频分析主链路（A/B/C/D/E）
- `POST /api/search`：仅检索（用于证据预览与调试）
- `POST /api/chat/stream`：流式问答（优先）
- `POST /api/chat`：非流式降级问答
- `GET /api/traces/{trace_id}`：问答与检索追踪回放

说明：若后端暂未完整提供上述接口，前端仍按该契约设计状态机与组件槽位，避免后续二次翻修布局；后端按该契约补齐实现。

## 4.4 后端融合架构（目标）

后端采用“编排层 + 能力层 + 可观测层”三层：

1. 编排层（`pipeline`）
   - 统一 A/B/C/D/E 阶段状态机
   - 负责异常中断、重试与降级路径（`chat/stream -> chat`）
2. 能力层（`processors/retrievers/llm/rag`）
   - `processors`：音频转写、视觉证据提取
   - `retrievers`：Dense + Sparse + RRF + Rerank
   - `llm`：OpenAI-compatible 客户端
   - `rag`：上下文构建与证据装配
3. 可观测层（`tracing`）
   - 生成 `trace_id`
   - 记录召回、融合、重排、上下文、回答片段

## 4.5 后端功能缝合映射（来自 Local-VQA）

- 检索：引入 Hybrid 检索链路（Dense/Sparse/RRF/Rerank）
- 问答：引入 RAG 上下文构建与流式回答
- 证据：引入 citation 统一结构（时间锚点、来源类型、文本摘要、图片路径）
- 追踪：引入 `trace_id` 全链路白盒追踪（默认 JSONL，可选 OTel）
- 兼容：保留历史 `query_template` 读取兼容，但新写入仅允许 `query_text`

## 4.6 Agent 框架引入决策

结论：**首期不将 Agent 框架作为硬依赖**。

原因：

1. 当前主目标是“功能缝合 + 稳定交付”，核心是确定性的 A/B/C/D/E 流程、Hybrid 检索、RAG 问答与可观测链路。
2. 这些能力可通过现有编排层 + 规则化状态机稳定实现，不需要先引入额外 Agent 运行时复杂度。
3. 在桌面端场景下，过早引入多 Agent 编排会增加调试和回归成本，拉长收敛周期。

保留扩展位（阶段 2 之后再评估）：

- 若后续要支持“多轮任务规划、自动策略切换、工具自主调用”，再引入轻量 Agent 层
- Agent 只放在“决策层”，执行层仍保持可回放、可审计的确定性 pipeline

## 6. 目标设计语言（Linear + Sentry + Raycast）

## 6.0 风格来源与比例

- 主风格（约 70%）：`Linear.app`  
  负责整体布局、层级、留白、排版节奏与工作台气质。
- 状态语义（约 20%）：`Sentry`  
  负责告警、失败、运行态、健康度等高密度状态表达。
- 命令交互（约 10%）：`Raycast`  
  负责命令面板、快捷动作、键盘优先交互与轻量浮层。

设计目标：保持“工程化、克制、可扫描”的桌面生产力体验，而不是品牌营销风格。

## 6.1 设计关键词

- Calm Precision（冷静精确）
- Layered Clarity（分层清晰）
- Soft Depth（柔和纵深）
- Data First（数据优先）

## 6.2 视觉基线

- 字体：优先 `Geist` / `-apple-system`，中文回退 `PingFang SC`
- 圆角：`10 / 14 / 18` 三级
- 栅格：8pt 基线系统
- 阴影：低对比、短半径，避免厚重卡片堆叠
- 模糊：仅用于顶栏/浮层（轻量，不喧宾夺主）
- 颜色：石墨中性色为主，强调色按语义分配（主操作/成功/警告/失败），避免单一品牌色覆盖全局
- 证据语义色：`audio` 使用冷青、`visual` 使用靛蓝、`ocr` 使用琥珀，统一出现在标签、时间锚点与引用边线

## 6.3 组件语义

- `Surface`：基础容器，承担层次，不承担强调
- `StageChip`：阶段状态最小单位（pending/running/completed/failed）
- `MetricPill`：耗时、日志条数、告警
- `ArtifactCard`：笔记/导图/字幕/导出入口
- `CommandBar`：全局动作（新建分析、导出、重跑、保存）
- `ModeTabs`：工作模式切换（流程分析 / 证据问答 / 检索调试）
- `AnswerStreamPane`：问答流式渲染容器（chunk 增量更新）
- `CitationCard`：证据卡（时间锚点、来源、摘要、置信度）
- `TimelineScrubber`：时间轴跳转（点击锚点联动播放器）
- `RetrievalDebugPanel`：Dense/Sparse/RRF/Rerank 结果并排对照
- `TraceDrawer`：trace_id 追踪侧抽屉（阶段耗时、配置快照、异常片段）
- `ImageLightbox`：原图放大查看（缩放、拖拽、关闭）

## 7. 创新排版与信息架构（主工作区）

采用“**双模式 + 三段式工作流剧场（Workflow Theater）**”布局：

1. 顶部 `Mode Tabs`（模式标签）
   - `流程分析`：强调 A/B/C/D/E 运行态与产物审阅
   - `证据问答`：强调 query -> retrieval -> answer -> citation
   - `检索调试`：强调 Dense/Sparse/RRF/Rerank 对照与参数试验
2. 左侧 `Stage Rail`（阶段轨道）
   - 固定展示 A/B/C/D/E 与关键子阶段
   - 始终可见当前阶段、异常阶段、可重试阶段
3. 中央 `Focus Canvas`（主内容焦点区）
   - 根据模式切换为“流程主视图”或“问答主视图”
   - 内容超出单屏时优先标签页与折叠分区，不直接堆叠成长页
4. 右侧 `Insight Dock`（洞察侧舱）
   - 显示核心指标、证据摘要、历史快切、导出动作
   - 在问答模式下优先展示引用证据与 trace 快捷入口

该布局兼顾“创新感”与“可理解性”：视觉上像专业桌面工具，逻辑上围绕“流程产出 + 证据问答”双核心任务。

## 7.1 流程分析模式排版

- 左侧：阶段轨道（A/B/C/D/E）+ 失败节点提示 + 一键重跑
- 中央：阶段内容标签（日志 / 产物 / 配置快照）
- 右侧：任务指标（耗时、命中数、告警）、导出卡片、历史快切

适用场景：用户先跑完整分析链路，确认阶段状态与基础产物质量。

## 7.2 证据问答模式排版

- 左侧：问答会话列表（按任务聚合）+ 最近问题
- 中央：回答流（流式 chunk）+ 引用证据卡 + 锚点点击跳转
- 右侧：证据时间轴 + 图片缩略图 + 原图放大入口 + trace 摘要

适用场景：用户围绕“问题”反复追问，需要快速核验证据而不是重新浏览全量日志。

## 7.3 检索调试模式排版

- 左侧：查询输入与检索参数（top_k、rerank 开关、融合策略）
- 中央：Dense/Sparse/RRF/Rerank 四列对照结果
- 右侧：命中可视化、片段合并结果、阶段耗时与异常提示

适用场景：研发与高级用户定位“命中不准、回答不稳、证据不足”等质量问题。

## 8. 交互模型

## 8.1 全局交互

- `Cmd/Ctrl + K`：命令面板（任务搜索、打开配置、导出）
- `Cmd/Ctrl + 1..5`：阶段快速切换（A/B/C/D/E）
- `Cmd/Ctrl + Shift + 1..3`：工作模式切换（流程分析 / 证据问答 / 检索调试）
- 拖拽文件到窗口任意区域即可创建任务

## 8.2 阶段态交互

- 运行中：中央显示阶段日志流 + 右侧显示实时指标
- 完成后：自动切换至“产物审阅模式”
- 失败时：阶段轨道高亮失败节点 + 右侧给出可执行修复动作

## 8.3 证据问答交互

- 问题输入仅保留自然语言，不引入模板选择器
- 优先走 `chat/stream`，失败自动降级 `chat`，并给出可见状态提示
- 回答中每个引用证据都可点击：时间锚点跳视频、图片缩略图开原图、trace_id 开追踪抽屉
- 支持连续追问，默认继承最近一次检索上下文与筛选条件

## 8.4 历史态交互

- 历史记录采用“可固定侧边栏 + 智能过滤”形态
- 任务切换时保留滚动位置和最近打开的内容页签

## 8.5 标签页与弹窗使用策略

- 适当使用标签页：同一任务上下文中的并列内容（例如“日志 / 产物 / 配置详情”）优先放在页面主区域标签页，不打断用户工作流。
- 弹窗用于短时、原子任务：确认删除、重命名、导出选项、快捷配置，不承载长文阅读、复杂编辑或多步骤流程。
- 当单一界面内容放不下时，优先采用“分区 + 标签页 + 折叠面板 + 二级路由”组合，不把所有扩展内容塞进弹窗。
- 同一时刻原则上只允许一个主弹窗，禁止弹窗套弹窗；复杂任务采用独立侧栏或独立页面。
- 所有弹窗必须支持键盘关闭（`Esc`）与明确主次操作按钮，避免用户被困在模态状态。
- 图片预览是唯一允许的高频弹窗，且必须保持轻量：默认单图、可关闭、可键盘操作。

## 9. Electron 安全与工程约束

必须执行：

1. `contextIsolation: true`
2. `nodeIntegration: false`
3. preload API 白名单（最小暴露）
4. 禁止 `remote` 模块
5. 对外部链接使用系统浏览器打开，禁止任意 webview 执行

补充：

- 日志落盘到 `app.getPath('userData')`
- 后端启动路径、参数、健康检查、退出清理由主进程托管
- API Key 仅通过环境变量读取，前端不落盘、不回显完整密钥
- 图片证据采用本地路径白名单访问，禁止任意路径穿透读取
- 问答输入统一使用自然语言问题

建议命令：

- `npx getdesign@latest add linear.app`
- 必要时对照 `sentry` 与 `raycast` 的 `DESIGN.md` 手工合并状态与命令交互规则（不要直接混贴整份文档）。

## 10.1 引入方式

1. 使用 Impeccable 的 Codex 技能包（或本地已安装技能）。
2. 将项目设计上下文集中维护在 `DESIGN.md`（主文档）；`.impeccable.md` 仅作为可选个人工作笔记。
3. 所有关键界面迭代前后执行固定命令链。

## 10.2 命令链（推荐工作流）

每个核心界面（流程分析、证据问答、检索调试、历史、配置）都走以下流程：

1. `/critique`：先做 UX 评审，确认信息层级问题
2. `/arrange` + `/typeset`：先修布局与文字层级
3. `/colorize`（轻量）+ `/normalize`：统一视觉令牌与对比度
4. `/polish`：出最终稿
5. `/audit`：做可访问性与技术质量校验

## 10.3 设计守则（避免“AI 套模板感”）

- 禁止“卡片套卡片”的冗余容器结构
- 禁止高饱和紫色渐变主导页面
- 禁止使用弱对比文本表达关键状态
- 必须突出阶段状态语义与任务主链路
- 必须突出“问题 -> 证据 -> 回答”的闭环，不得把证据信息藏在二级弹窗深处

## 10.4 Local-VQA 功能逻辑缝合规范（前端）

前端状态机按以下顺序组织：

1. 任务创建后触发 `analyze`，进入 A/B/C/D/E 阶段流
2. 阶段 `D` 完成后，自动解锁“证据问答模式”与“检索调试模式”
3. 用户在问答模式发起问题后，优先走 `chat/stream`，失败降级 `chat`
4. 回答中的 citation 统一映射到时间轴与图片证据
5. 点击 `trace_id` 进入追踪抽屉，展示 query、召回、融合、重排、上下文摘要

前端数据对象至少包含以下字段：

- `query_text`：用户自然语言问题
- `trace_id`：一次问答的唯一追踪标识
- `citations[]`：证据数组（`source`, `start`, `end`, `text`, `image_path`）
- `retrieval_debug`：Dense/Sparse/RRF/Rerank 的中间结果摘要

GUI 排版约束：

- 回答正文、证据卡、时间轴必须同屏可见，避免用户在多弹窗间往返
- 检索调试结果必须支持表格与卡片双视图切换，兼顾扫描效率与细节阅读
- 原图预览保持轻量弹窗，不承载额外业务编辑逻辑

## 10.5 Local-VQA 功能逻辑缝合规范（后端）

后端状态机按以下顺序组织：

1. `analyze` 触发 A/B/C：媒体校验、转写、视觉处理
2. 进入 D：并行执行 Dense 与 Sparse 召回，执行 RRF 融合与 Rerank
3. 进入 E：按 `rag` 配置构建上下文，调用 `chat/stream` 或 `chat`
4. 生成统一 citation 并返回给前端（含 `source/start/end/text/image_path`）
5. 记录 `trace_id` 全链路日志，支持 `/api/traces/{trace_id}` 回放

后端必须落地的契约与兼容规则：

- 统一输入：仅接受 `query_text` 作为新写入字段
- 历史兼容：允许读取 `query_template` 并映射到统一 `query_text` 语义
- 统一错误语义：流式中断、检索为空、模型不可用必须返回可判定错误码
- 统一降级策略：流式失败自动回退非流式，不改变最终 citation 结构

后端性能与可观测要求：

- 对 Dense/Sparse/RRF/Rerank 分别记录耗时与命中统计
- 追踪日志默认落盘 JSONL，支持按配置开启 OpenTelemetry
- 证据图片保留原图路径，不在 API 层做二次压缩重编码

## 12. 验收标准（Definition of Done）

功能与体验必须同时满足：

1. 桌面端可独立启动并拉起后端服务
2. 全流程可用：`A/B/C/D/E` 阶段、问答、回放、导出
3. 阶段状态在任意时刻都能一眼识别，且支持 `Cmd/Ctrl + 1..5` 快速切换
4. 主要页面通过 `Impeccable + 技术审计` 双重校验
5. UI 风格统一，符合“Linear 主风格 + Sentry 状态语义 + Raycast 命令交互”的克制与高可读原则
6. 问答链路支持流式输出，并在异常时自动降级非流式，不丢失可见状态反馈
7. 回答区具备证据闭环：时间锚点可跳转、图片可放大、引用可追踪 `trace_id`
8. 检索调试模式可查看 Dense/Sparse/RRF/Rerank 核心结果，不依赖后端日志排查
9. 问题输入仅为自然语言，界面中不存在模板配置与模板回填入口
10. 用户不需要理解前后端分离细节即可完成主要任务
11. 后端实际执行 Hybrid 检索链路，且可返回融合前后结果摘要
12. 后端 `trace_id` 回放可覆盖 query、召回、融合、重排、上下文、回答片段
13. 历史兼容策略生效：旧 `query_template` 仅兼容读取，新写入只包含 `query_text`
14. 流式与非流式回答在 citation 结构上保持一致，前端可无差别渲染

## 13. 风险与回滚策略

主要风险：

- Electron 与后端进程协同复杂度提升
- 现有前端逻辑迁移期间的行为回归
- 视觉重构过度导致学习成本上升
- 双模式与多面板引入后信息密度过高，导致认知负担增加
- 证据图片与流式渲染并发下可能出现渲染卡顿
- Hybrid 检索参数不稳可能导致回答波动
- 历史兼容处理不当可能导致旧任务回放异常

对应策略：

1. 保留旧 Web 前端分支作为应急回退
2. 每阶段提供可运行里程碑，避免“大爆炸上线”
3. 对核心流程建立自动化回归（任务创建、SSE、证据跳转、导出）
4. 对视觉重构做小范围可用性验证后再全量替换
5. 证据问答模式默认“渐进披露”：先显示回答与关键引用，调试信息按需展开
6. 对长会话列表与大图预览做虚拟化/懒加载策略，降低渲染负担
7. 固化 Hybrid 检索参数基线并建立离线评测集，避免参数漂移
8. 为历史任务建立兼容回放测试集，发布前执行全量抽检

## 15. 结论

该方案可以在不推翻后端能力的前提下，将 VidGnost 升级为面向生产场景的桌面工作台，并平滑吸收 `Local-VQA` 的问答与证据能力。  
核心价值不在“更炫”，而在“更可控、更可读、更高效”：让用户在同一桌面工作台里同时完成流程分析与证据问答闭环。
