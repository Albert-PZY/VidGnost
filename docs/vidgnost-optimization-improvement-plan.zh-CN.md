# VidGnost 本地单机优化改进方案（精简版）

## 1. 文档信息

- 版本：v2.0（本地单机精简）
- 日期：2026-04-07
- 适用范围：`VidGnost` 当前单用户、单机、本地化使用场景
- 目标：在不增加无关复杂度的前提下，提升稳定性、可维护性、易用性和结果质量

---

## 2. 约束与原则

本方案严格遵循以下前提：

1. 产品定位：本地单机工具，不做在线多用户平台
2. 模型来源：第三方在线 API + 本地 Whisper 转写
3. 优化边界：不引入复杂基础设施，不做超前架构
4. 目标导向：优先解决“经常遇到、直接影响使用体验”的问题

不纳入本方案的方向：

- 平台化治理体系
- 多账号与协作体系
- 数据库化改造

---

## 3. 当前基线与主要问题

## 3.1 当前可用能力

VidGnost 已具备完整闭环：

- URL / 路径 / 上传的视频输入
- A/B/C/D 阶段化执行
- 本地 `faster-whisper-small` CPU 转写
- 在线 LLM 生成结构化笔记与导图
- SSE 实时日志与阶段进度
- 历史任务回放与导出

## 3.2 当前最值得优化的问题（按优先级）

### P0：必须优先解决

1. 核心编排文件过大，维护负担高
- `backend/app/services/task_runner.py` 职责过多，改动风险集中

2. 历史任务读取在任务量增长后会变慢
- 当前文件扫描式读取对小规模够用，但条目多后会有明显下降

3. 前端主入口状态仍偏重
- `frontend/src/App.tsx` 承担过多状态协调，后续加功能容易回归

### P1：建议次阶段解决

1. API 地址硬编码，不利于本地多环境切换
2. 事件日志高频写盘，长任务时 I/O 压力偏高
3. 配置命名仍有历史残留，影响长期维护理解

### P2：可选优化

1. 长视频任务的恢复与重试体验还可增强
2. 模板与导出体验还可进一步做“少操作、快复用”

---

## 4. 优化目标（90 天）

1. 稳定性：长任务失败率下降，失败后更容易恢复
2. 可维护性：核心模块可拆分、可测试、可替换
3. 易用性：减少用户重复操作，首跑成功率提升
4. 结果质量：输出内容一致性和可读性更稳

---

## 5. 完整优化方案（本地单机版）

## 5.1 后端编排重构（P0）

### 目标

把“能跑”升级为“可持续迭代”。

### 改进措施

1. 拆分 `task_runner` 职责
- `pipeline_orchestrator`：阶段状态流转
- `stage_executor_*`：A/B/C/D 实际执行
- `runtime_event_service`：事件与日志发布
- `artifact_persistence_service`：阶段产物与快照持久化

2. 统一错误分类与失败恢复入口
- 归一化常见失败类型（输入、下载、转写、生成、导出）
- 保证任务失败后仍可拿到完整诊断信息

3. 保持文件存储方案，但增加轻量索引文件
- 不引入新的存储中间件
- 使用增量维护的索引 JSON（如 `tasks/index.json`）加速列表检索

### 验收标准

1. `task_runner.py` 行数下降 40% 以上
2. 关键路径单元测试覆盖提升
3. 历史任务列表在本地大样本下响应明显改善

---

## 5.2 运行稳定性优化（P0/P1）

### 目标

减少“任务跑到一半异常”与“异常难定位”。

### 改进措施

1. 事件落盘优化
- 由“每事件立即落盘”改为“短周期批量 flush + 终态强制落盘”

2. 下载与准备流程增强
- Whisper 模型下载支持更明确的重试与状态提示
- 下载失败信息精简成可操作提示（网络、磁盘、文件校验）

3. 长任务健康检查
- 在任务中按阶段记录关键健康指标（耗时、分块数、重试次数）
- 明确“可继续运行告警”和“必须终止错误”边界

### 验收标准

1. 长视频任务稳定性提升
2. 常见问题定位时间明显缩短
3. 失败日志对用户可读、对开发可诊断

---

## 5.3 前端可维护性与交互优化（P1）

### 目标

降低前端回归风险，提升操作顺滑度。

### 改进措施

1. 继续拆分 `App.tsx`
- 将任务流状态、配置状态、面板状态进一步解耦
- 主入口仅保留页面编排

2. 配置与文案一致性清理
- 清理与当前 CPU-only、API-only 不一致的历史命名
- 统一“用户看到的字段名”和“后端实际行为”

3. 历史与导出体验细节
- 常用导出动作前置
- 历史任务过滤和排序更直观

### 验收标准

1. 前端主入口体量进一步降低
2. 新增功能改动范围更可控
3. 用户完成一次分析的交互步骤减少

---

## 5.4 结果质量改进（P1/P2）

### 目标

让输出更稳定、更可用、更便于二次编辑。

### 改进措施

1. 提示词模板质量基线
- 为常见场景提供内置模板（通用总结、课程整理、访谈提炼），当前默认模板不改动
- 统一模板输出结构，减少随机性

2. 阶段 D 输出一致性校验
- 基础结构校验（标题层级、关键段落存在性）
- 异常输出自动回退到更稳健的生成策略（同模型同通道）

3. 导出前轻量检查
- 保证笔记/导图/字幕字段完整，避免导出空文件

### 验收标准

1. 输出格式一致性提升
2. 用户二次编辑工作量下降
3. 导出成功率和可用率提升

1. 后端工程师 1 名（主导编排重构与稳定性）
2. 前端工程师 1 名（主导状态拆分与交互优化）
3. 测试/联调 1 名（可兼职）

---

## 8. 风险与控制

1. 风险：重构期间引入回归
- 控制：分阶段灰度 + 回归用例先行

2. 风险：过度优化导致迭代变慢
- 控制：只做问题驱动，不做超前设计

3. 风险：本地环境差异导致稳定性波动
- 控制：自检项与错误提示继续标准化

---

## 9. 成功判定

满足以下条件可判定本轮优化成功：

1. 端到端任务成功率提升
2. 长任务失败率下降
3. 故障定位时间缩短
4. 代码可维护性显著提升（核心文件复杂度下降）
5. 用户完成分析与导出的体验更顺畅

---

## 10. 备注

本方案聚焦单机本地实践，不引入无关复杂系统。后续如产品定位发生变化（例如多人协作或云端部署），再单独制定扩展版方案。

## 11. 执行状态（2026-04-07）

本轮已按本方案完成落地，核心结果如下：

1. 后端编排与持久化职责拆分
- 已新增 `task_artifact_persistence_service.py`，统一运行告警与阶段产物持久化。
- 已新增 `runtime_event_service.py`，统一阶段事件、日志、子阶段状态与阶段指标落盘。
- `task_runner.py` 已改为编排主流程，事件与指标逻辑委派到独立服务。

2. 失败诊断与恢复入口统一
- 已新增 `task_error_classifier.py`，将常见失败归类为输入/下载/转写/生成/导出/运行时。
- 任务失败事件已附带 `category` 与 `hint`，便于前端与日志直接定位问题。

3. 历史列表性能优化
- 已引入 `backend/storage/tasks/index.json` 增量索引。
- `TaskStore.list()` 已改为“索引筛选 + 分页加载记录文件”模式，避免全量扫描。

4. 运行稳定性优化
- EventBus 事件落盘已改为批量 flush 与终态强制落盘，减少高频 I/O。
- Whisper small 模型下载失败信息已转换为可操作提示（网络/磁盘/校验方向）。
- 阶段指标已补充分块数、运行锁等待、纠错模式、回退状态等健康指标。

5. 前端可维护性与配置体验
- `VITE_API_BASE_URL` 已接入前端 API 客户端，支持本地多环境切换。
- `App.tsx` 继续拆分：快速开始文档加载与主视图 props 组装已下沉到独立模块。
- 历史遗留命名已清理：`normalizeWhisperConfigForCpu` 统一 CPU-only 语义。

6. 结果质量与导出可用性
- 已提供预置模板：`default/course/interview`（summary + mindmap）。
- 阶段 D 输出已加入一致性校验与 fallback 兜底策略。
- 导出接口已增加轻量完整性检查，避免导出空内容。

- `uv run --project backend python -m compileall backend/app` 通过。
