# VidGnost 前端 Electron 全量重构方案（Linear + Sentry + Raycast）

> 状态：方案文档（仅规划，不涉及代码变更）  
> 适用范围：`F:\in-house project\VidGnost`  
> 目标：将现有 Web 前端重构为 Electron 桌面应用，并完成视觉与信息架构的彻底改版

---

## 1. 重构目标

本次重构目标不是“把网页套个壳”，而是完成以下三类升级：

1. 运行形态升级：从浏览器工作台升级为桌面原生工作台（Electron）。
2. 交互与视觉升级：全面替换当前布局、排版、组件风格，建立“Linear 主风格 + Sentry 状态语义 + Raycast 命令交互”的设计语言。
3. 产品表达升级：以“视频分析主流程（A/B/C/D + 子阶段）”为信息中心，构建更直观、更高信任感的任务驱动 UI。

---

## 2. 范围与边界

### 2.1 本次重构范围

- 前端运行容器：Vite Web -> Electron（主进程 + 渲染进程 + preload）
- UI 信息架构：页面结构、导航模式、面板布局、交互路径
- 视觉系统：色彩、字体、间距、组件样式、动效规范
- 前端工程组织：模块分层、状态流、IPC 封装、桌面能力接入

### 2.2 明确不在本次范围

- 后端业务逻辑（FastAPI 的转写与阶段执行逻辑）
- 历史数据结构强制迁移（仅做兼容，不破坏存量）
- 模型算法层重写

---

## 3. 当前前端架构摘要（重构基线）

当前项目是 React + Vite + TypeScript Web 应用，主入口和核心视图集中在：

- `frontend/src/App.tsx`
- `frontend/src/components/workbench-*.tsx`
- `frontend/src/hooks/use-*.ts`

核心能力已具备：

- SSE 实时任务流
- 阶段日志与产物展示
- 配置面板 / 历史任务 / 导出链路

当前主要问题（从桌面产品目标看）：

1. 布局仍偏“网页面板拼接”，非桌面生产力应用心智。
2. 信息密度高时，视觉层级不够稳定（阶段、日志、产物争夺注意力）。
3. 窗口级能力缺位（原生菜单、系统级快捷键、文件拖放体验、窗口状态管理）。
4. 前端与后端的本地运行关系由用户手工维护，不符合“桌面应用一体化交付”预期。

---

## 4. 目标架构（Electron）

## 4.1 进程分层

- `main` 进程：窗口、菜单、协议、生命周期、自动更新、后端守护进程管理
- `preload`：受控 IPC 桥，白名单 API 暴露给渲染层
- `renderer`：React UI（纯展示与交互，不直接触碰 Node 能力）

## 4.2 后端协同模式（推荐）

采用“桌面壳 + 本地后端服务”模式：

1. Electron 启动时检测 `http://127.0.0.1:8000/health`
2. 未就绪则由主进程拉起后端（开发态 `uv run`，生产态打包可执行）
3. 就绪后渲染层使用本地 API（保持当前接口协议不变）

优势：最小化后端改造，保留现有任务与存储体系。

## 5. 桌面工程结构（目标）

```text
frontend-desktop/
├─ electron/
│  ├─ main/                 # BrowserWindow / app lifecycle / backend bootstrap
│  ├─ preload/              # contextBridge + typed IPC
│  └─ shared/               # IPC contract, constants
├─ renderer/
│  ├─ src/
│  │  ├─ app/               # layout shell, routing, providers
│  │  ├─ features/          # task-runtime, history, config, export
│  │  ├─ components/        # design-system + business components
│  │  ├─ styles/            # tokens, themes, motion
│  │  └─ lib/
└─ build/                   # installer config, icons, signing config
```

说明：保留现有 `frontend` 代码可读资产，但重构后建议迁移到 `renderer/features/*`，避免继续以单体 `App.tsx + hooks` 扩张。

## 6. 目标设计语言（Linear + Sentry + Raycast）

## 6.0 风格来源与比例

- 主风格（约 70%）：`Linear.app`  
  负责整体布局、层级、留白、排版节奏与工作台气质。
- 状态语义（约 20%）：`Sentry`  
  负责告警、失败、运行态、健康度等高密度状态表达。
- 命令交互（约 10%）：`Raycast`  
  负责命令面板、快捷动作、键盘优先交互与轻量浮层。

设计目标：保持“工程化、克制、可扫描”的桌面生产力体验，而不是品牌营销风格。

## 6.1 设计关键词

- Calm Precision（冷静精确）
- Layered Clarity（分层清晰）
- Soft Depth（柔和纵深）
- Data First（数据优先）

## 6.2 视觉基线

- 字体：优先 `Geist` / `-apple-system`，中文回退 `PingFang SC`
- 圆角：`10 / 14 / 18` 三级
- 栅格：8pt 基线系统
- 阴影：低对比、短半径，避免厚重卡片堆叠
- 模糊：仅用于顶栏/浮层（轻量，不喧宾夺主）
- 颜色：石墨中性色为主，强调色按语义分配（主操作/成功/警告/失败），避免单一品牌色覆盖全局

## 6.3 组件语义

- `Surface`：基础容器，承担层次，不承担强调
- `StageChip`：阶段状态最小单位（pending/running/completed/failed）
- `MetricPill`：耗时、日志条数、告警
- `ArtifactCard`：笔记/导图/字幕/导出入口
- `CommandBar`：全局动作（新建分析、导出、重跑、保存）

## 7. 创新排版与信息架构（主工作区）

采用“三段式工作流剧场（Workflow Theater）”布局：

1. 左侧 `Stage Rail`（阶段轨道）
   - 固定展示 A/B/C/D 与 D 子阶段
   - 一眼判断“当前卡在哪一步”
2. 中央 `Focus Canvas`（主内容焦点区）
   - 当前阶段主内容（转写流 / 融合提示词 / 详细笔记）
   - 内容超出单屏时优先使用页内标签页（如“预览 / 编辑 / 对照 / 原文”）分层，不直接堆叠成长页
   - 强制保持一个主任务焦点，避免信息噪音
3. 右侧 `Insight Dock`（洞察侧舱）
   - 关键指标、历史快切、产物卡片、操作按钮
   - 保留高价值快捷操作，不打断中间阅读流

这个排版兼顾“创新感”与“可理解性”：视觉上更像专业桌面工具，逻辑上始终围绕主流程。

## 8. 交互模型

## 8.1 全局交互

- `Cmd/Ctrl + K`：命令面板（任务搜索、打开配置、导出）
- `Cmd/Ctrl + 1..4`：阶段快速切换
- 拖拽文件到窗口任意区域即可创建任务

## 8.2 阶段态交互

- 运行中：中央显示流式文本 + 右侧显示实时指标
- 完成后：自动切换至“产物审阅模式”
- 失败时：阶段轨道高亮失败节点 + 右侧给出可执行修复动作

## 8.3 历史态交互

- 历史记录不再只是列表弹窗，改为“可固定侧边栏 + 智能过滤”
- 任务切换时保留滚动位置和最近打开的内容页签

## 8.4 标签页与弹窗使用策略

- 适当使用标签页：同一任务上下文中的并列内容（例如“日志 / 产物 / 配置详情”）优先放在页面主区域标签页，不打断用户工作流。
- 弹窗仅用于短时、原子任务：确认删除、重命名、导出选项、快捷配置，不承载长文阅读、复杂编辑或多步骤流程。
- 当单一界面内容放不下时，优先采用“分区 + 标签页 + 折叠面板 + 二级路由”组合，不把所有扩展内容塞进弹窗。
- 同一时刻原则上只允许一个主弹窗，禁止弹窗套弹窗；复杂任务改为独立侧栏或独立页面。
- 所有弹窗必须支持键盘关闭（`Esc`）与明确主次操作按钮，避免用户被困在模态状态。

## 9. Electron 安全与工程约束

必须执行：

1. `contextIsolation: true`
2. `nodeIntegration: false`
3. preload API 白名单（最小暴露）
4. 禁止 `remote` 模块
5. 对外部链接使用系统浏览器打开，禁止任意 webview 执行

补充：

- 日志落盘到 `app.getPath('userData')`
- 后端启动路径、参数、健康检查、退出清理由主进程托管

## 10. 使用 Impeccable 辅助视觉设计的执行方案

参考仓库：`https://github.com/pbakaus/impeccable`
本地已安装相关skills

## 10.0 设计基线文件（DESIGN.md）接入

结合 `awesome-design-md` 使用方式，先落地主风格基线：

1. 在项目根目录放置 `DESIGN.md`（建议以 `linear.app` 为基础）。
2. 在 `DESIGN.md` 中补充本项目特有规则：阶段状态色、日志密度、导出区层级、桌面窗口行为。
3. 让 AI 在每次 UI 生成/重构时同时读取 `DESIGN.md + AGENTS.md + .impeccable.md`。

建议命令：

- `npx getdesign@latest add linear.app`
- 必要时对照 `sentry` 与 `raycast` 的 `DESIGN.md` 手工合并状态与命令交互规则（不要直接混贴整份文档）。

## 10.1 引入方式

1. 使用 Impeccable 的 Codex 技能包（或本地已安装技能）。
2. 将项目设计上下文集中维护在 `.impeccable.md`（当前仓库已存在）。
3. 所有关键界面迭代前后执行固定命令链。

## 10.2 命令链（推荐工作流）

每个核心界面（工作台、历史、配置）都走以下流程：

1. `/critique`：先做 UX 评审，确认信息层级问题
2. `/arrange` + `/typeset`：先修布局与文字层级
3. `/colorize`（轻量）+ `/normalize`：统一视觉令牌与对比度
4. `/polish`：出最终稿
5. `/audit`：做可访问性与技术质量校验

## 10.3 设计守则（避免“AI 套模板感”）

- 禁止“卡片套卡片”的冗余容器结构
- 禁止高饱和紫色渐变主导页面
- 禁止使用弱对比文本表达关键状态
- 必须突出阶段状态语义与任务主链路

## 12. 验收标准（Definition of Done）

功能与体验必须同时满足：

1. 桌面端可独立启动并拉起后端服务
2. 现有任务全流程（创建、运行、回放、导出）可用
3. 阶段状态在任意时刻都能一眼识别
4. 主要页面通过 `Impeccable + 技术审计` 双重校验
5. UI 风格统一，符合“Linear 主风格 + Sentry 状态语义 + Raycast 命令交互”的克制与高可读原则
6. 用户不需要理解前后端分离细节即可完成主要任务

---

## 13. 风险与回滚策略

主要风险：

- Electron 与后端进程协同复杂度提升
- 现有前端逻辑迁移期间的行为回归
- 视觉重构过度导致学习成本上升

对应策略：

1. 保留旧 Web 前端分支作为应急回退
2. 每阶段提供可运行里程碑，避免“大爆炸上线”
3. 对核心流程建立自动化回归（任务创建、SSE、导出）
4. 对视觉重构做小范围可用性验证后再全量替换

## 15. 结论

该方案可以在不推翻后端能力的前提下，将 VidGnost 升级为面向生产场景的桌面工作台。  
核心价值不在“更炫”，而在“更可控、更可读、更高效”：让用户把注意力放在视频分析主流程，而不是界面操作本身。
