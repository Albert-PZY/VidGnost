# VidGnost 项目完整优化方案与执行清单

更新时间：2026-04-11

## 1. 前言：当前项目分析结论（原文整理版）

### 1.1 总体判断

我看了当前代码结构、主视图实现，并在本地跑了一次前端预览。项目的底子其实不错：产品定位清晰，核心能力链也不弱，本地转写、VQA、自检、导出、换肤、桌面壳层这些都已经成型。按当前阶段，我会给它一个大致评价：

- 视觉完成度：`7/10`
- 性能基础：`7/10`
- 功能实用性：`7.5/10`
- 创新表达：`6/10`

最大的短板不是“没有功能”，而是“能力没有被前端充分转化成用户感知价值”。现在更像一个完成度不错的专业工具壳层，还没完全长成一个让人记住的“证据驱动视频分析工作台”。

### 1.2 视觉层面

- 当前视觉方向基本符合你自己的设计定位文档，走的是冷静、专业、深色科技工具路线，这点是成立的，基线没有跑偏。
- 但页面之间的“身份感”还不够强。新建任务、历史记录、设置中心、系统自检都大量复用同一种 `Card + 标题 + 描述 + 分组列表` 结构，统一是统一了，但记忆点不够，容易显得像同一个模板换文案。
- 首屏“新建任务”现在更像上传器，而不是价值展示页。用户一进来先看到上传框和流程步骤，但看不到“笔记整理最终长什么样”“VQA 会如何给证据和时间点”，所以首屏缺少转化力。建议把首屏改成“输入区 + 结果样张 + 两种工作流差异说明”的结构，而不是只放一个大拖拽框。
- 任务结果页的信息价值很高，但视觉表达偏平。当前笔记和思维导图区域还是直接用纯文本或 `pre` 展示，结构层级、可读性、专业感都被压扁了。这里应该升级成真正的 Markdown 渲染视图，支持标题层级、列表、引用、Mermaid 图片、时间锚点和证据块。
- 任务处理页现在是固定 `50/50` 双栏，专业感有，但空间利用不够聪明。大屏下用户会希望自己决定视频区和结果区谁更大。你已经有可调整分栏依赖和封装，但没真正用起来，这是很可惜的。
- 视觉上最该强化的，不是继续堆皮肤，而是把“证据”“时间点”“可追溯”“本地优先”做成界面语言。也就是让时间轴、片段、命中来源、trace 身份，比边框和卡片更有存在感。

### 1.3 性能层面

- 你已经做了合理的首屏懒加载和空闲预热，方向是对的。
- 当前最明显的性能热点不是首屏，而是编辑器包体。前面构建里 `prompt-markdown-editor` chunk 已经到 `1MB+`，这对 Electron 也不是小数。你虽然已经 lazy load 了编辑器，但设置中心里仍然会承受比较重的解析和渲染成本。可以继续拆成“只在打开提示词弹窗时再加载编辑器内核”，而不是进入设置页就准备这套重量级能力。
- 依赖层面还有可减重空间。`recharts` 依赖和 chart 封装还在，但当前业务页面没有充分使用。如果确认没用，直接删掉能减一点包体和维护噪音。
- 历史页现在一次拉 `100` 条，而且没有分页或虚拟列表。任务少时没问题，但长期 Electron 使用、积累上百上千任务后，这里会先变成渲染和交互卡顿点。
- 当前 VQA 前端仍然走非流式问答，这会直接拖低“感知性能”。而文档又把“实时回复”作为既定能力写进去了，这个反差很大。
- 我本地预览时，在后端不可用的情况下，前端会出现对 `/api/tasks/recent`、`/api/tasks/stats`、`/api/config/ui` 的连接拒绝错误，但界面没有清晰的“后端启动中/连接失败/降级模式”壳层状态。这是桌面应用很实际的问题。建议做一个明确的 shell bootstrap 状态机，而不是只靠 toast。

### 1.4 功能实用性

- 这是当前最值得优化的部分。后端和文档里已经有不少高级能力，但前端没有把它们完整交付出去。
- 文档里已经把 `flow / qa / debug` 三种模式、流式问答、Trace 回放都当成目标能力写清楚了，但当前任务处理页只有一个简化版问答区，没有 debug 模式，也没有 trace 追溯入口。
- 自检的“自动修复”能力也存在于接口层，但 UI 没有真正交给用户。这类“能修不能点”的能力很浪费。
- 任务控制也类似。取消任务和重跑 D 阶段 API 已经有了，但处理页没有做成显眼的运行控制。这对于长任务、失败重试、结果纠偏都很关键。
- 历史记录页的实用性还可以再上一个台阶。文档里写的是支持 `workflow/status` 过滤，但当前只有工作流过滤，没有状态过滤。这会让“找失败任务、找处理中任务、找未完成任务”变得低效。
- 提示词模板现在是 CRUD 完成了，但“可用性”还没到位。列表预览只是 3 行截断文本。更实用的形态应该是：模板变量预览、样本文本 dry-run、版本 diff、当前生效范围和最近使用效果。
- 首屏新建任务从当前主界面来看，用户主要只感知到文件上传入口，但产品文档里强调过 URL、Path、Upload 三种来源。这个“多源输入”的价值值得重新显化，不然用户会把产品误解成“本地文件转写器”，而不是“视频分析工作台”。

### 1.5 创新点还能怎么做

- 最有潜力的创新方向，不是再加一个功能，而是把已经有的多模态和 trace 能力变成“别人一眼记住”的交互。
- 第一条建议是做“证据时间轴”。把转写片段、VQA 命中、关键帧、笔记锚点放到同一条时间轴上，点击答案时同时联动视频、转写和笔记。这样“多模态证据驱动”就不是一句说明文案，而是一个真正可感知的界面资产。
- 第二条建议是做“Trace Theater”。既然后端已经有 trace、重排、检索链路，就不该只把结果给用户，而应该给一个“为什么得到这个答案”的可视化视图，让用户看到 dense/sparse/rerank、命中来源、提示词、最终回答之间的关系。这会让产品在可信度上明显拉开差距。
- 第三条建议是做“跨任务研究板”。允许用户把多个任务中的片段、笔记、关键帧拖进一个研究工作区，形成知识板或素材池。这对学习、研究、内容生产都很实用，而且很符合“长视频信息提取与复盘”的定位。
- 第四条建议是做“边处理边出价值”。不是等任务全部结束再给结果，而是在转写、纠错、总结过程中连续吐出“阶段性摘要”“关键片段”“待确认问题”。这会显著缩短时间到价值的感知。
- 第五条建议是做“Prompt Lab”。让同一段转写在两个模板、两个模型、两种纠错模式之间并排对比。这会把设置中心从配置页提升成一个真正能驱动效果优化的专业工具。

### 1.6 我会怎么排优先级

- 第一优先级：把已存在但未交付完整的能力补齐。
  包括流式 VQA、trace 入口、自检 auto-fix、任务取消/重跑 D、历史状态过滤。
- 第二优先级：把结果页从“文本容器”升级成“知识工作区”。
  包括 Markdown 真预览、Mermaid/图片/时间锚点、可调分栏、结果联动视频。
- 第三优先级：把桌面产品的运行态做实。
  包括启动态壳层、后端连接状态、降级模式、历史分页或虚拟化、无后端恢复路径。
- 第四优先级：再做差异化创新。
  包括证据时间轴、Trace Theater、跨任务研究板、Prompt Lab。

## 2. 执行原则

本方案默认遵循以下原则：

- 不为了“看起来更新”而做表面式改动，所有改动必须提升用户可感知价值。
- 优先补齐后端已具备、但前端尚未充分交付的能力。
- 优先处理会影响长期桌面使用体验的问题，再做风格层面的增强。
- 所有视觉优化都必须服务于“证据驱动、可追溯、本地优先、专业高效”这四个关键词。
- 所有性能优化都要有明确指标，不做纯经验主义“微优化”。

## 3. 完整优化方案

## 3.1 P0：能力补齐与核心体验修复

目标：

- 让产品的真实能力和前端呈现能力对齐。
- 先补齐最影响实用性的缺口，避免“文档里有、界面里没有”的落差。

范围：

- 任务处理页
- VQA 问答链路
- 自检页
- 历史页

交付项：

- 接入流式 VQA 回答，支持 chunk 实时显示、done 收束、error 回退。
- 在任务处理页加入 `trace_id` 可见入口，并支持查看一次问答的检索链路和命中依据。
- 在任务处理页加入“取消任务”和“重跑 D 阶段”控制区。
- 在系统自检页加入“自动修复”按钮、修复过程状态和结果反馈。
- 历史页补齐状态过滤，并支持快速筛选失败、处理中、已完成任务。
- 新建任务页补齐 URL、Path、Upload 三种输入入口，让“多源输入”重新可见。

验收标准：

- 用户在 VQA 模式下输入问题后，1 秒内能看到首段流式内容开始输出。
- 每次问答结束后，界面可看到 `trace_id`，并能查看关键命中依据。
- 正在运行的任务可以取消，失败或异常终态任务可以直接触发重跑 D 阶段。
- 自检发现可修复问题时，用户可在页面上直接点击修复并看到结果。
- 历史页可以通过 `workflow + status + search + sort` 组合定位任务。

## 3.2 P1：结果页升级为知识工作区

目标：

- 把当前“查看结果”的页面升级成“操作知识资产”的页面。
- 让结果页成为产品核心价值最强的地方。

范围：

- 任务处理页
- 笔记和思维导图展示区
- 视频联动区

交付项：

- 笔记和思维导图从纯文本切换为 Markdown 真渲染。
- 支持 Mermaid 图片、标题层级、列表、引用块、代码块、图片证据块。
- 支持时间戳锚点点击后联动视频跳转。
- 将固定 `50/50` 双栏改成可拖拽分栏。
- 强化视频、转写、笔记、问答结果四者的联动关系。
- 为转写片段增加“引用到笔记”“作为 VQA 证据提问”“加入研究板”的快捷动作。

验收标准：

- 用户可以在任务处理页中顺畅阅读 Markdown 结果，而不是只能看纯文本。
- 视频播放、转写片段、笔记锚点、问答命中之间可以双向跳转。
- 大屏幕用户可以自由调节视频区和内容区的宽度。

## 3.3 P1：桌面壳层与运行态体验补强

目标：

- 让 Electron 使用体验更像成熟桌面产品，而不是“网页装进壳里”。

范围：

- 应用启动
- 后端连接
- 全局状态提示
- 长列表与长期运行性能

交付项：

- 增加启动阶段状态机：
  - 正在初始化
  - 正在连接后端
  - 后端不可用
  - 降级模式
  - 运行正常
- 增加“重试连接”“查看诊断信息”“打开日志目录”等恢复动作。
- 历史页引入分页或虚拟列表。
- 设置中心和任务处理页中重型组件继续按需加载。
- 增加基础性能监控面板或开发态性能日志开关。

验收标准：

- 后端不可用时，用户不会只看到 toast 和空页面，而是看到明确状态和可执行操作。
- 历史记录在 1000+ 任务量级下滚动和筛选仍保持可接受流畅度。
- 设置中心首开时间和提示词编辑弹窗首开时间可被量化跟踪。

## 3.4 P2：视觉体系强化

目标：

- 让项目从“完成度不错的工具界面”升级成“有辨识度的专业产品”。

范围：

- 首页
- 结果页
- 历史页
- 自检页
- 设置中心

交付项：

- 首屏改版为“任务输入 + 价值样张 + 两类工作流差异说明”。
- 为不同页面建立更强的页面身份：
  - 新建任务强调入口与结果预期
  - 历史页强调检索与回放
  - 自检页强调诊断与修复
  - 设置页强调实验与调优
- 弱化模板化卡片堆叠，增强关键内容块的层级与节奏。
- 将“证据”“时间点”“本地优先”“可追溯”做成视觉母题。
- 减少只有装饰价值的通用分隔和重复容器，把视觉注意力拉回任务价值。

验收标准：

- 新用户进入首页 10 秒内能理解两类工作流的差异和产出结果。
- 用户能仅凭页面观感区分“任务页 / 历史页 / 设置页 / 自检页”，不再觉得只是同一模板换标题。

## 3.5 P2：产品差异化创新

目标：

- 建立 VidGnost 的差异化记忆点，而不是停留在“另一个 AI 工具”。

范围：

- 多模态证据协同
- Trace 可视化
- 多任务复盘
- 提示词实验

交付项：

- 证据时间轴：
  - 统一承载转写片段、VQA 命中、关键帧、笔记锚点。
- Trace Theater：
  - 展示 Dense / Sparse / RRF / Rerank / 最终答案之间的关系。
- 跨任务研究板：
  - 支持用户把不同任务中的片段、证据、笔记卡片拖入一个集合空间。
- Prompt Lab：
  - 支持模板 A/B 对比、输出对比、不同模型策略对比。
- 阶段性价值输出：
  - 在任务尚未完全结束前，持续产出关键摘要、重点片段、可继续追问的问题。

验收标准：

- 用户不需要阅读文档，也能在界面中直接感知“证据驱动”和“可追溯”是产品核心能力。
- 至少有 1 个功能能成为对外展示时的明显亮点，而不是只靠口头介绍。

## 6. 实施时需要同步关注的风险

- 流式 VQA 接入后，需要统一处理 chunk、done、error、fallback，避免前端状态机混乱。
- Markdown 真渲染接入后，要控制滚动同步、锚点跳转和大文本渲染性能。
- 可拖拽分栏会改变任务页布局，需要同步考虑小屏幕和低分辨率桌面环境。
- 历史虚拟化会影响搜索、定位、滚动恢复逻辑，需要先定义好列表交互模型。
- Prompt Lab 和跨任务研究板属于高价值创新，但复杂度较高，必须放在基础体验稳定后实施。

---

## 7. 建议量化指标

为了避免“改完了但不知道是否更好”，建议为每个阶段设指标：

### 7.1 体验指标

- 新用户首次完成任务创建耗时
- 用户从进入首页到看到首个有效结果的时间
- 用户在 VQA 中从提问到看到首段回答的时间
- 用户定位历史失败任务的平均操作步数

### 7.2 性能指标

- 设置中心首次打开耗时
- 提示词编辑弹窗首次打开耗时
- 历史页 1000 条任务下滚动帧率
- Electron 长时间运行 2 小时后的内存稳定性

### 7.3 价值指标

- 问答中证据点击率
- trace 查看率
- 自动修复触发率与修复成功率
- 导出功能使用率
- 跨任务研究板使用率

---

## 8. 最终结论

VidGnost 当前最需要的，不是继续增加零散功能，也不是只做视觉美化，而是：

- 先把已经存在的后端能力，完整、明确、可信地交到前端手里。
- 再把任务结果页升级成真正的知识工作区。
- 然后把 Electron 桌面运行态体验做扎实。
- 最后用证据时间轴、Trace Theater、跨任务研究板、Prompt Lab 这些能力，建立清晰的产品记忆点。

如果按这套路线执行，VidGnost 会从“一个完成度不错的本地视频分析工具”，逐步变成“一个真正专业、可信、可追溯、且有产品辨识度的多模态视频分析工作台”。
